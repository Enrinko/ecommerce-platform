- **Дата:** 2026-07-07
- **Статус:** черновик на ревью
- **Автор:** Max
- **Тип документа:** roadmap (источник правды по всему проекту). Каждая фаза далее получает свой отдельный спек и план реализации.

---

## 1. Что это

Портфолийный, но «взрослый» по инженерии проект: **интернет-магазин** со сквозным стеком —
веб-витрина, админ-панель, мобильное приложение и единый backend API. Цель — на одном
цельном продукте показать полный цикл full-stack-разработки на современном стеке
(React / Next.js / Nest.js / Postgres / MongoDB / Docker / CI).

Проект осознанно построен как **монорепозиторий с несколькими приложениями и общими пакетами**,
чтобы показать не только «умею писать фичи», но и умение вести архитектуру, контракты и инфраструктуру.

## 3. Чего НЕ делаем (non-goals)

Осознанно ограничиваем объём, чтобы держать качество:

- Не строим реальный приём денег — Stripe только в **test mode**, деньги не двигаются.
- Не делаем мультивендорность/маркетплейс — один продавец, один каталог.
- Не делаем сложную систему промо/скидок в фазе 1 (базовые цены; скидки — опциональное расширение).
- Не делаем i18n/мультивалютность на старте (одна локаль, одна валюта; заложим точки расширения).
- Не делаем собственную систему доставки/логистики — статусы заказа моделируем, интеграций с ТК нет.
- Не гонимся за паритетом «пиксель-в-пиксель» между вебом и мобилкой — общий домен, разный UI.

## 4. Архитектура

### 4.1. Структура монорепо

```
fullstack/
├─ apps/
│  ├─ api/          # Nest.js — REST API, источник бизнес-логики
│  ├─ web/          # Next.js — витрина для покупателя
│  ├─ admin/        # Next.js — админ-панель (отдельное приложение)
│  └─ mobile/       # Expo (React Native) — мобильная витрина
├─ packages/
│  ├─ types/        # zod-схемы + производные TS-типы — единый контракт
│  ├─ api-client/   # типизированный клиент к API (fetch-обёртки)
│  ├─ ui/           # общие UI-примитивы для web/admin (Tailwind + shadcn)
│  ├─ eslint-config/# общий ESLint-конфиг
│  └─ tsconfig/     # базовые tsconfig
├─ docker-compose.yml   # postgres + mongo + (опц.) api
├─ turbo.json           # пайплайны сборки/тестов
├─ pnpm-workspace.yaml
├─ package.json
└─ docs/
   └─ superpowers/specs/  # этот roadmap + пофазные спеки/планы
```

### 4.2. Приложения и их границы

- **`apps/api` (Nest.js)** — единственное место с бизнес-логикой и доступом к БД. Всё остальное — клиенты.
- **`apps/web`** — витрина: каталог, товар, корзина, checkout, авторизация, история заказов.
- **`apps/admin`** — управление: CRUD товаров/категорий, заказы и их статусы, пользователи, дашборд.
- **`apps/mobile`** — витрина покупателя на RN: каталог, товар, корзина, checkout, заказы.

### 4.3. Общие пакеты и принцип «единого контракта»

Ключевое архитектурное решение: **`@repo/types` — единый источник правды по контрактам**.
zod-схемы описывают формы запросов/ответов и доменные сущности. Из них выводятся TS-типы (`z.infer`).

- Nest использует эти схемы для валидации входящих данных (через zod-pipe).
- `@repo/api-client` типизирует запросы/ответы теми же типами.
- Формы на фронте (`react-hook-form` + `@hookform/resolvers/zod`) валидируются теми же схемами.

Одно изменение контракта → рассинхрон ловится компилятором во всех приложениях сразу.

- **`@repo/api-client`** — «глупые» типизированные функции над API (fetch). React-query-хуки живут
  в приложениях (у web/admin/mobile разная конфигурация кэша/SSR), но поверх одного клиента.
- **`@repo/ui`** — общие примитивы для web и admin. Мобилка (RN) не переиспользует UI (другие
  примитивы), но переиспользует `types` и `api-client`.

### 4.4. Поток данных (пример: оформление заказа)

```
web/mobile (react-hook-form + zod)
      │  POST /orders  (тело валидируется zod-схемой из @repo/types)
      ▼
apps/api (Nest) ── ZodValidationPipe ──► OrdersService
      │                                      │
      │                                      ├─ Prisma: транзакция (заказ + позиции, списание стока)
      │                                      ├─ PaymentProvider.createIntent() (Mock | Stripe test)
      │                                      └─ событие "order.created"
      ▼
ответ (typed) ──► react-query инвалидация кэша ──► UI обновился
```

## 6. Модель данных (обзор)

Детали (поля, индексы, связи) уточняются в спеке Фазы 1. Здесь — обзор.

### 6.1. PostgreSQL (Prisma)

- **User** — id, email, passwordHash, role (`customer` | `admin`), createdAt.
- **Category** — id, name, slug.
- **Product** — id, title, slug, description, priceCents, currency, stock, categoryId, images[], isActive.
- **Cart / CartItem** — корзина пользователя и её позиции (productId, qty). (Гостевая корзина — на клиенте.)
- **Order** — id, userId, status (`pending`→`paid`→`shipped`→`delivered`/`cancelled`), totalCents, адрес доставки, createdAt.
- **OrderItem** — orderId, productId, titleSnapshot, priceCentsSnapshot, qty. (Снимок цены/названия на момент заказа.)
- **Payment** — orderId, provider, providerRef, status, amountCents.

Цены храним в **минорных единицах (cents)** целыми числами — без ошибок float.
В `OrderItem` — снимок названия и цены, чтобы история заказа не «плыла» при изменении товара.

### 6.2. MongoDB (Mongoose)

- **Review** — productId, userId, rating (1–5), title, body, createdAt.
  Отдельно — денормализованная агрегация рейтинга товара (avg, count), пересчёт при новом отзыве.

Обоснование: отзывы — высокочастотные, слабосвязанные, с гибким телом; классический кейс документной БД.

## 7. Сквозные решения

- **Auth:** JWT access-токен (короткий TTL) + refresh-токен в httpOnly-cookie. Роли `customer`/`admin`,
  guard на уровне Nest. Пароли — argon2. Защита админских ручек ролью.
- **Валидация и ошибки:** единый формат ошибки API `{ statusCode, message, errors? }`.
  Входные данные валидируются zod-схемами из `@repo/types`. Глобальный exception-filter в Nest.
- **API-дизайн:** REST, версионирование префиксом `/api/v1`, пагинация `?page&limit`,
  единый конверт списков `{ items, total, page, limit }`. OpenAPI-документация из декораторов.
- **Платежи:** интерфейс `PaymentProvider { createIntent, handleWebhook }`. Локально — `MockProvider`
  (сразу «оплачено» без ключей). В демо — `StripeProvider` (test mode + обработка вебхука).
- **Безопасность-минимум:** helmet, rate-limit на auth-ручках, CORS по списку origin, секреты через env.

## 8. Дорожная карта по фазам

Каждая фаза = отдельная ветка, свой спек, свой план реализации, свой PR. Ниже — цель и критерии готовности.

### Фаза 1 — Фундамент + Backend API ← **старт**
**Цель:** запускаемый, покрытый тестами API, на который затем сядут все клиенты.
**Состав:** монорепо (pnpm + Turborepo), пакеты `types`/`api-client`/конфиги; Nest-приложение;
Prisma-схема + миграции; модуль отзывов на Mongo; auth (JWT+роли); домен (товары, категории,
корзина, заказы); Swagger; docker-compose (Postgres+Mongo); GitHub Actions (lint/typecheck/test/build).
**Готово, когда:** `docker-compose up` поднимает БД; `pnpm --filter api test` зелёный;
Swagger доступен; сиды создают демо-каталог и админа; CI проходит на PR.

### Фаза 2 — Веб-витрина (Next.js)
**Цель:** покупатель может пройти путь «каталог → товар → корзина → checkout → заказ».
**Состав:** каталог с фильтрами/пагинацией, страница товара с отзывами, корзина (zustand),
checkout (react-hook-form + zod), авторизация, история заказов, react-query поверх `api-client`.
**Готово, когда:** сквозной сценарий покупки проходит в Playwright; адаптив mobile-first; lint/typecheck зелёные.

### Фаза 3 — Админ-панель (Next.js)
**Цель:** админ управляет каталогом и заказами.
**Состав:** CRUD товаров/категорий, список и смена статусов заказов, список пользователей,
простой дашборд (кол-во заказов/выручка), защита ролью `admin`.
**Готово, когда:** админ создаёт товар, он виден на витрине; смена статуса заказа отражается в истории покупателя.

### Фаза 4 — Мобильное приложение (React Native / Expo)
**Цель:** паритет витрины покупателя на мобилке.
**Состав:** навигация, каталог, товар, корзина, checkout, авторизация, заказы — поверх `api-client` и `types`.
**Готово, когда:** сборка Expo запускается; сквозной сценарий покупки работает на эмуляторе.

## 9. Стратегия тестирования

- **API:** юнит-тесты сервисов (бизнес-логика: расчёт суммы заказа, списание стока, переходы статусов)
  + e2e через supertest (реальные ручки на тестовой БД).
- **Веб:** Vitest на утилиты/хуки; Playwright на сквозной сценарий покупки.
- **Контракты:** типизация от `@repo/types` — компилятор ловит рассинхрон клиент/сервер.
- **CI-гейт:** PR не мёржится без зелёных lint + typecheck + test.

Приоритет тестирования — **бизнес-логика с ветвлениями** (суммы, сток, статусы, платежи), а не геттеры.

## 10. DevOps

- **Локально:** `docker-compose up` — Postgres + Mongo (+ опц. API). `.env.example` в каждом приложении.
- **Образы:** многостадийные Dockerfile для `api` и `web` (slim-рантайм).
- **CI (GitHub Actions):** на PR — install (кэш pnpm) → lint → typecheck → test → build (Turborepo-кэш).
  Отдельный job — сборка docker-образов на push в main.
- **Окружения:** конфигурация только через env; секреты не коммитим; `*.env.example` документируют переменные.
- **Деплой (вне объёма реализации, зафиксировано как вариант):** web → Vercel, api → Railway/Render/Fly,
  БД — управляемые инстансы. Не входит в критерии готовности фаз.
